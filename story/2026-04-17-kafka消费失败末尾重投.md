# Kafka 消费失败末尾重投

## 背景

Kafka 同一 topic 的同一 partition 内消息是串行推进的。当前 Kafka 消费失败后直接 `nack` 延迟重试，会阻塞该 partition 后续消息继续消费。对于事务消息场景，这会放大单条异常消息对后续消息吞吐的影响。

## 目标

- 仅针对 Kafka 消费场景，在业务消费失败时将消息重新投递到 topic 末尾。
- 当前失败消息应完成 `ack`，避免继续阻塞同一 partition 的后续消息。
- 重投消息作为一条新的事务消息保存并派发，不新增表字段，使用 `parentId` 记录原失败消息 id。

## 范围

- 调整 Kafka 消费监听器的失败处理逻辑。
- 通过 `MessagePublishService` 重新发布 Kafka 消息。
- 增加与 Kafka 末尾重投相关的测试。

## 非目标

- 本次不对 RabbitMQ 或其他 MQ 引入相同策略。
- 本次不新增重试来源字段、重试次数字段或新的数据表字段。
- 本次不调整消息顺序语义，接受 Kafka 末尾重投带来的顺序变化。

## 关键设计

- 仅对“业务消费失败/事务回滚”场景做末尾重投。
- 对消息头解析失败、payload 反序列化失败等场景，仍保持当前 `nack` 逻辑，不直接重投。
- Kafka 业务消费失败时：
  - 构造新的 `TransactionalMessage`
  - `destination` 使用当前 topic
  - `route` 使用当前 record key
  - `headers` 继续透传
  - `parentId` 通过 `sendWithParent(MqType.KAFKA, ...)` 续接为原消息 id
  - `rootId` 沿用原链路 rootId
  - `id` 重新生成
- 只有当新消息保存成功后，才 `ack` 当前失败消息；若重投保存失败，则继续 `nack`

## 验收标准

- Kafka 业务消费失败时，会通过 `MessagePublishService` 重发一条新消息到 topic 末尾。
- 新消息使用新 id，`parentId` 指向原失败消息 id。
- 当前失败消息在重投成功后完成 `ack`，不会继续卡住 partition。
- 解析失败场景仍保持原有 `nack` 行为。
- 与本次改动直接相关的 Kafka 测试通过。

## 风险与阻塞

- 该方案会改变 Kafka 同一业务链路的原始顺序，失败消息将排到 topic 末尾。
- 若重投构造逻辑处理不当，可能导致消息头或链路字段丢失。
