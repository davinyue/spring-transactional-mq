# 派发调度器归属调整

## 背景

当前 `TransactionalMessageDispatchScheduler` 位于 `spring-transactional-mq-spring-boot-starter` 模块，并直接依赖 `TransactionalMqProperties`。这会导致事务消息的后台派发调度能力绑定在 Spring Boot starter 上，不利于 `core` 模块作为 Spring 事务消息核心能力独立存在。

## 目标

- 将事务消息后台派发调度能力下沉到 `spring-transactional-mq-core`。
- 去掉调度器对 starter 配置对象的直接依赖，改为显式构造参数。
- 保持 starter 侧自动装配能力不变，只负责参数绑定与生命周期托管。

## 范围

- 调整 `TransactionalMessageDispatchScheduler` 所在模块和包位置。
- 调整 starter 自动配置中的调度器装配方式。
- 增加与本次调整直接相关的单元测试和集成测试。

## 非目标

- 本次不调整成功消息归档与消费记录清理调度器的模块归属。
- 本次不重构 `MessagePublishService` 的职责边界。
- 本次不改动对外 API 语义和消息派发逻辑。

## 关键设计

- `TransactionalMessageDispatchScheduler` 迁移到 `core.service` 包下，作为核心派发能力的一部分。
- 调度器不再依赖 `TransactionalMqProperties`，仅通过构造函数接收 `dispatchBatchSize`、`dispatchIdleSleepMillis` 和 `MessageDispatchWakeupService`。
- starter 继续保留 `TransactionalMqProperties`，并在自动配置时将调度参数映射到调度器构造函数。
- `MessagePublishService` 保持“落库 + 事务提交后唤醒”的职责，不在内部创建或管理后台派发线程。

## 验收标准

- `core` 模块可以直接使用 `TransactionalMessageDispatchScheduler`，无需依赖 starter 配置类。
- starter 自动装配后仍能获得事务消息派发调度能力。
- 相关测试可以覆盖调度器核心行为和 starter 装配链路。
