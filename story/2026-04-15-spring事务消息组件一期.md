# Spring 事务消息组件一期

## 背景

当前需要在空项目基础上实现一个面向 Spring Boot 的事务消息组件业务系统通过统一发送接口先将消息落库，再由异步派发线程发送到 MQ；消费者通过统一接口消费消息，并依赖数据库记录实现消费幂等项目要求兼容 JDK 1.8、Spring Boot 2.7.18，优先以 Spring Boot Starter 方式集成，存储层使用 ez-mybatis，并优先支持 RabbitMQ，同时为后续扩展 RocketMQ、Kafka 保留统一抽象

## 目标

- 提供可落库、可异步派发、可幂等消费的事务消息主链路
- 提供 starter 形式的自动装配能力，降低业务接入成本
- 提供 RabbitMQ 发送与消费能力，支持多实例部署和多线程消费
- 提供真实 Oracle + RabbitMQ 环境测试能力，验证核心链路
- 支持消息链路追踪、历史归档、发送日志和消费历史审计

## 范围

- 父工程与多模块 Maven 结构
- common/api/core/store-ezmybatis/rabbitmq/starter/demo 模块
- BaseEntity 与审计字段
- 事务消息主表、历史表、发送日志表、已消费主表、已消费历史表
- RabbitMQ 生产发送、消费者自动注册、手动 ack、多线程消费
- 发送链路 `parentId/rootId`、消费上下文完整 headers、消息重发链路
- 默认 JSON 序列化、RabbitMQ gzip 传输、真实环境测试

## 非目标

- 本期不实现 RocketMQ、Kafka 具体适配器
- 本期不引入自定义注解驱动消费者
- 本期不内置完整调度中心或可视化管理界面
- 本期不清理或迁移历史 `tasks/` 目录内容，后续由用户自行处理

## 关键设计

- 使用单一需求文件沉淀整次开发，具体执行项只体现在本文件任务清单中
- 发送接口语义收敛为“消息入库”，`TransactionalMessageSender.send(...)` 直接返回消息 `id`，不再返回 `SendResult`
- `MessagePublishService` 实现发送接口，并在发送方法上增加事务注解
- 消息存储使用 ez-mybatis DSL，不保留手写 SQL 仓储逻辑
- `TXN_MESSAGE` 作为派发状态主表，支持多实例下批量查询候选、按 `id` 稳定排序、逐条条件抢占、立即发送 MQ、按 `id + dispatchToken` 批量更新最终状态
- `TXN_MESSAGE_HISTORY` 作为成功消息归档表；`TXN_MESSAGE_SEND_LOG` 保存发送流水；`TXN_CONSUMED_MESSAGE` 与 `TXN_CONSUMED_MESSAGE_HISTORY` 负责去重与历史审计
- 消息链路增加 `parentId`、`rootId`，并提供 `sendWithParent(message, parentContext)`，只续接链路，不复制业务内容
- RabbitMQ 消费者接口直接定义 `getQueueName()/getMinConcurrency()/getMaxConcurrency()`，由自动注册器为每个消费者创建 `SimpleMessageListenerContainer`
- `ConsumeContext` 除快捷字段外，补充完整 `headers`，保证业务消费者可拿到原始消息头
- 默认 `payload_text` 入库存原始 JSON 文本；RabbitMQ 发送时对 JSON 文本做 gzip 压缩，消费时 gunzip 后再反序列化
- 历史表表名统一为 `TXN_MESSAGE_HISTORY`

## 验收标准

- 能通过发送接口将消息写入数据库，并返回消息 `id`
- 派发线程能从数据库查询候选消息、抢占发送并写入发送日志
- RabbitMQ 生产端发送内容为 gzip 压缩字节，消费端可自动解压后反序列化
- 消费者支持多线程消费，并通过自动注册器完成容器注册
- 消费前会记录幂等信息，重复消息不会重复执行业务消费
- 真实 Oracle + RabbitMQ 测试可通过，覆盖发送、消费、归档、日志、历史链路
- `parentId/rootId`、完整 headers 能贯通发送、MQ、消费上下文、历史表和发送日志
