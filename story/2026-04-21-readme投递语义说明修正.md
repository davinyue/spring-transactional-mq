# README 投递语义说明修正

## 背景

当前 README 将组件语义直接描述为“至少一次投递”，容易让业务接入方理解为同一条消息可能重复调用业务消费逻辑。实际消费侧会先写入消费记录表，并通过唯一约束判断是否已经消费过；重复消息会被框架拦截，不再调用业务系统的消费逻辑。

## 目标

修正 README 中关于投递语义和消费幂等的表述，区分“生产到 MQ 链路可能重复投递”和“消费侧业务逻辑是否重复执行”。

## 范围

- 调整 README 适用场景中的重复投递说明。
- 调整 README 投递语义章节。
- 调整 README 当前限制中的相关说明。

## 非目标

- 不修改运行时代码。
- 不调整消费记录表结构、清理逻辑或 MQ ack/nack 行为。
- 不补充新的自动化测试。

## 关键设计

README 保留生产链路可能重复投递的事实，同时明确消费侧在调用业务消费者前会先写消费记录；消费记录已存在时，框架确认消息并跳过业务消费逻辑。对消费记录清理后的极旧重复消息识别限制，作为剩余边界明确说明。

## 验收标准

- README 不再笼统地把整体语义描述为业务消费逻辑“至少一次执行”。
- README 明确生产到 MQ 链路可能重复投递。
- README 明确消费侧通过消费记录表拦截重复消息。
- README 说明消费记录清理后的边界风险。
