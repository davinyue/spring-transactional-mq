# 发送线程批量候选逐条抢占批量落状态

## 背景

多实例发送线程如果查询出交叉候选消息，需要避免在一个事务中按不确定顺序批量持有多行锁。新的发送流程改为先批量查询候选，再在内存中按 `id` 排序，逐条条件抢占，抢占成功后立即发送 MQ，最后按 `id + dispatchToken` 批量落最终状态。

## 需求

- 批量查询候选消息，不在查询阶段抢占。
- 候选消息在发送服务中按 `id` 稳定排序。
- 单条条件更新抢占消息，抢占失败返回空并忽略。
- 抢占成功写入 `dispatchToken`，最终成功/失败更新必须按 `id + dispatchToken` fencing。
- 一批消息处理完成后，统一提交成功集合和失败集合的最终状态。
- 更新 SQL、实体、映射和测试。
